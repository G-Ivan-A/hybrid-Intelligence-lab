## Гипотеза

Когда AI-исполнитель получает GitHub issue **без контекста предыдущих задач**, он
должен по одной формулировке определить (1) тип артефакта и (2) режим выполнения
(Creative / Structured / Hybrid). Гипотеза: минимальный, прозрачный набор правил,
привязанный к **действию** (глаголу/deliverable), а не к теме задачи,
воспроизводит человеческую разметку типов и режимов на реальных задачах Хаба —
и его ошибки указывают на объективно недоспецифицированные формулировки, а не на
слабость правил.

## Метод

`classify.py` — единственный артефакт-эксперимент. Он реализует два
классификатора и пять тестов:

- **v1 (naive keyword-vote)** — каждое ключевое слово голосует с равным весом;
  baseline.
- **v2 (action-anchored)** — сначала удаляются мета-перечисления типов
  («Research/Analysis/Audit» как ссылка на саму систему типов), затем побеждает
  первое сработавшее *действие* в детерминированном порядке приоритетов; тема
  без действия голоса не даёт. Если действие не найдено — формулировка помечается
  `under-specified`, и v2 честно падает обратно на наивный голос.

Ground truth берётся из человеческой разметки: типовой префикс задачи в бэклоге
(Test 1) или фактически выполненный артефакт практического issue (Test 2).
Классификатор **никогда** не видит этот префикс — он снимается перед подачей на
вход, что и моделирует «работу без контекста».

| Тест | Что проверяет | Датасет |
| --- | --- | --- |
| Test 1 | Классификация типа/режима задач бэклога | B-016..B-039 (18 задач) |
| Test 2 | Интерпретация реальных issue без контекста | #310, #316, #320, #322, #324 |
| Test 3 | Разрешение конфликта «творчество vs строгий контракт» | 3 сконструированные задачи |

| Test 5 | Согласованность наших режимов с индустриальными рамками | 6 задач × Cynefin/Bloom |

  промах (B-016) помечен `under-specified` — в формулировке нет действия.

## Границы применимости

Это **исследовательский зонд**, не продакшн-классификатор и не предложение нового
контракта. Правила выведены из конвенций маршрутизации этого репозитория и
намеренно подогнаны под наблюдаемую разметку, чтобы измерить *достижимый потолок*
классификации по минимальному тексту и локализовать граничные случаи. Обобщающая
точность на несмежных задачах не заявляется. Интерпретация результатов — в
родительском отчёте.
